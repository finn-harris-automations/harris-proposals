Shift · Events Extraction

From one Claude doing everything, to a pipeline where each step earns its place.

A reset of the events ingestion build — moving from a single agentic loop that breaks at scale, to a staged pipeline where the LLM only does the one thing it’s genuinely best at.

Prepared for Don & Carl · Shift Prepared by Finn Harris · Harris Automations Date 2026-05-06
Where we are

The current build is hitting an architectural ceiling, not a prompt one.

A single Claude call is being asked to search the web, fetch pages, parse them, translate, categorise, and emit JSON — all inside one agent loop. The symptoms (token-limit errors, malformed JSON, low yield, runaway cost) all trace back to that single decision.

Symptoms

What you’re seeing today

Token-limit errors that needed max_tokens raised from 4k to 20k. “role: assistant should not be last message” failures. Inconsistent JSON shapes between runs. Roughly 20 usable events surfaced from 300 source sites — below 7% conversion.

Underlying cause

The LLM is the orchestrator, not a leaf

Every page goes through the same agent loop deciding what to fetch, how to parse it, what counts as on-topic, and how to structure the output — in one call. Each round-trip compounds the chance of a malformed response. The cost and brittleness are emergent properties of that shape.

The decision

Invert the architecture.

Stop asking Claude to be the orchestrator. Make it the cleanest possible leaf node — one prompt, one page, one JSON output — and let purpose-built tools handle everything else.

Today

One agent doing six jobs

  • Decides which URL to crawl next
  • Fetches HTML and extracts content
  • Translates Dutch ↔ English
  • Categorises and tags each event
  • Scores topical relevance
  • Emits final JSON
→
After

Ten small steps, one of them an LLM

  • Source registry routes by site type
  • Static fetchers pull HTML — no LLM
  • Embeddings filter on-topic vs off
  • Haiku 4.5 extracts events — the only LLM call
  • DeepL translates on write
  • Confidence gate decides publish vs review
How it flows

The new pipeline, end to end.

Ten stages, each doing one thing well. Only one stage is an LLM call — and it’s the stage where an LLM is genuinely irreplaceable. Everything else is a deterministic tool, a vector lookup, or a native API.

1

Source registry

Postgres

One Postgres table holds every event source — URL, site type (API / aggregator / static), success rate, last-scanned timestamp. Bad sources auto-deprioritise over time.

2

Dispatcher

BullMQ

Routes each source down the right track based on its type. Handles retries, rate limits, and concurrency. Already runs on your existing Heroku Redis — no new infrastructure.

3

Three-track fetch

API · HTML · JSON-LD

API-first where possible (Eventbrite / Meetup / Luma). Per-domain HTML adapters for big aggregators. JSON-LD schema.org/Event sniff for static venue sites — that alone covers 30–70% of pages without any LLM call at all.

4

Topical pre-filter

Embedding cosine

Before paying for any LLM token, check whether the page is even about sustainability. A Cohere multilingual embedding compared against a small reference set of on-topic Dutch text. ~250× cheaper than asking Claude to make the same call.

5

Per-page extractor

Haiku 4.5 — the one LLM call

Page markdown goes in. Zero or more structured event objects come out. One call per page. No agent loop. No web access. No tools. Forced structured output guarantees a valid JSON shape every time.

6

Categoriser

Embedding cosine

Each extracted event is matched against your existing category list using embeddings — not the LLM. Deterministic, ~1000× cheaper, and impossible to hallucinate a category that doesn’t exist in your DB.

7

Translator

DeepL

Dutch ↔ English on write. Purpose-built for this exact task and roughly 50× cheaper than asking an LLM. Decoupling translation from extraction means we can re-translate the back catalogue without re-running extraction.

8

Geocoder

PDOK Locatieserver

Free, authoritative Dutch geocoding endpoint. Turns “Amsterdam, Plantage Middenlaan 2A” into {postal_code, city} matching your EventItem.address shape. No hallucinated postcodes.

9

Deduplication

Composite key + pgvector

The same event listed on three sites still becomes one row. Composite key on normalised title + date bucket + city handles the easy 90%; embedding similarity cleans up the long tail.

10

Confidence gate

Threshold logic

The extractor emits a self-reported confidence score with each event. The gate decides what reaches your published feed automatically, what queues for human review, and what gets dropped. This is where the cost-of-review savings live — see next section.

Source strategy

Match the tool to the surface.

Not every event source needs the same treatment. Three tracks, and the LLM only fires on the third.

Track A

API-backed

Native APIs — zero LLM

Eventbrite, Meetup, Luma, Mobilizon all expose structured event data with native filters. We pull, filter, and write. The LLM never touches these.

When it’s the right fit Sources that already have a structured API endpoint with sustainability-relevant categories or geo filters.
Track B

Aggregator

Per-domain HTML adapter

Big neighbourhood platforms (Wijkconnect, Buurtgroen020, HalloIJburg) have predictable HTML. ~30 minutes per site to write a cheerio adapter, then it runs deterministically forever — with an LLM fallback if selectors break.

When it’s the right fit High-volume sources where the structure is stable enough that a one-off adapter pays back across hundreds of events.
Track C

Static venue

JSON-LD → LLM fallback

Long-tail venue sites. We try schema.org/Event JSON-LD first — it’s already structured data, no LLM needed. Only if the page doesn’t have it does Haiku 4.5 do the structured extraction.

When it’s the right fit The 7 hardcoded URLs in the current build. This is the track Phase 1 ships, and the only one that touches an LLM at all.
The toolkit

Each component picked for the job it’s actually good at.

Nine pieces total. Only one of them is a generative model. The rest are purpose-built — cheaper, faster, and impossible to hallucinate.

Claude Haiku 4.5 Extraction

Cheapest top-tier model with strong structured-output. The single place an LLM earns its tokens: turning unstructured page text into a typed event object.

Cohere embed-multilingual-v3 Topical filter + categoriser

Trained on 100+ languages with explicit cross-lingual alignment. Avoids the English-bias trap that English-only embeddings hit on Dutch sustainability vocabulary.

DeepL Translation

Best-in-class for NL ↔ EN, ~50× cheaper than running an LLM as a translator. Idiom handling on community Dutch (buurtmoestuin, energiecoöperatie) is meaningfully better than any general LLM.

PDOK Locatieserver Geocoding

Official Dutch government geocoding. Free, no API key, authoritative postcodes. Eliminates the “Claude invented a Dutch postcode” failure mode entirely.

cheerio HTML parsing

Server-side jQuery for HTML. Fast, well-known, zero magic. Used for per-domain aggregator adapters and JSON-LD sniffing.

BullMQ + Redis Job queue

Already running on your Heroku stack. Handles retries, scheduling, concurrency limits. No new infrastructure to deploy.

pgvector Dedup & similarity

Postgres extension for vector similarity. Lets us catch “same event, different listing” without an extra service to operate.

Postgres + TypeORM Storage

Already your stack. The new pipeline writes to the same EventItem table you have today — with a few small schema fixes (single image URL, structured address, nullable recurrence).

Heroku Hosting

Already your stack. The pipeline ships as additional NestJS modules in your existing app — not a separate service.

Confidence gate

The highest-leverage decision in the build.

The hidden cost of an events pipeline isn’t API tokens — it’s human review time. Every extracted event sits at one of three confidence levels, and each level has a different cost-of-failure trade-off.

≥ 0.85
Auto-publish

Goes live without review

The extractor is highly confident every required field is correct. These flow straight to your public feed.

0.6–0.85
Queue for review

Human eye before publish

Borderline confidence. Quick approve / reject in an internal queue. Cheaper than hand-curating, safer than auto-publishing.

< 0.6
Drop silently

Not worth a human’s time

Very low confidence. Logged for source quality metrics, but never queued. Bad sources surface in the registry as low yield.

Why this matters: at $30–40/hr review time and 30 seconds per event, hand-reviewing 100 events per run costs $25–35 in human time — far more than the LLM. Auto-publishing the high-confidence band is the single biggest cost lever in the system.

The line in the sand

What the LLM never does — on purpose.

The single biggest reason v0 broke is that the LLM was asked to do work it isn’t the right tool for. The build plan draws a hard line: each of the below stays out of the LLM call, forever.

Out of scope for the LLM

If a future contributor proposes pushing any of these back into the prompt, the build is being un-done. Each is delegated to a purpose-built tool that’s cheaper, faster, and can’t hallucinate.

×Translation. DeepL handles NL ↔ EN.
×Categorisation. Embedding cosine vs your category list.
×Topical relevance. Embedding pre-filter handles it.
×Geocoding. PDOK Locatieserver, authoritative.
×Deduplication. Composite key + pgvector.
×Tagging or theming. Your DB has no tags column.
×Web search or fetch. Static fetchers run before the LLM call.
×Multi-step reasoning. No agent loop, no tool round-trips.

Everything the LLM doesn’t do is something a deterministic tool does better. That’s the whole bet.
