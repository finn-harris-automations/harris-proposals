Where we are

From agentic guessing to deterministic extraction

The current build burns ~300k tokens per run for ~20 events at $1.50, with token-limit errors and inconsistent JSON. The architecture is inverted — Claude is orchestrating instead of leaf-extracting.

Today's pipeline

  • Single agent loop does search + fetch + parse + translate + categorize in one call
  • Token-limit errors, malformed JSON, inconsistent runs across the same sources
  • ~$1.50 for ~20 events; no per-source visibility, no failure attribution

With Automation

  • Single forced-tool Haiku 4.5 call per page, schema-locked to EventItem
  • Five-stage pipeline: fetch, parse, extract, normalize, write — each independently measurable
  • Per-URL extraction rate, cost, and p95 latency captured against a written contract
How it works

Two services, one clean pipeline

The agent loop is replaced by two well-bounded services inside shift-server. Each has a single job, a single contract, and clear failure attribution when something breaks.

EX

Per-page extractor

Fetches the source page, parses to markdown, then fires a single forced-tool Haiku 4.5 call with the EventItem schema as a tool input. No agent loop, no orchestration, no tool retries — one round-trip per page, every time.

AN Anthropic Haiku 4.5 CH cheerio MD node-html-markdown
NW

Normalizer and DB writer

Aligns extractor output to the EventItem entity. Parses date and time strings into Date objects, splits free-text addresses into postal_code and city via PDOK, picks one image url from the candidates, writes through TypeORM with translation rows in NL and EN.

TO TypeORM PG Postgres PD PDOK Locatieserver
